## 1. 总体结论

当前 PRD 整体方向正确，已经能够作为 **SnareLab V1.0 产品设计文档的基线版本**。

文档已经较好地覆盖了以下关键内容：

- 产品定位
- MVP 边界
- 非目标范围
- 响应式布局策略
- 核心页面与模块
- 数据模型草案
- 技术方案
- 分阶段开发计划

不过，在正式进入编码前，建议先进行一次 **V1.1 修订**，重点补强以下内容：

1. 收紧 MVP 范围，避免首页、日志和回顾系统过早膨胀。
2. 强化 `RhythmPattern` 数据模型，使其能够支撑播放、VexFlow 渲染、后续 MIDI 评分和套鼓扩展。
3. 明确节奏型数据主源，避免 TypeScript Catalog 与 JSON Catalog 分叉。
4. 新增 `PracticeExercise` 或类似实体，承接练习库中的可复用练习。
5. 为核心模块补充验收标准，提升开发可执行性。

最终判断：

> **建议通过产品方向评审，但在开发 Phase 0 前先完成 V1.1 小修版。**

## 3. 做得好的地方

### 3.1 产品定位清晰

PRD 已经将 SnareLab 定位为：

> 面向个人练习的架子鼓节奏训练系统。

这是正确方向。

它没有把产品写成单纯的“教学辅助系统”，也没有把产品降级成“节拍器增强版”。当前定位能够覆盖用户真实场景：

- 上课后回家练习；
- 自学节奏型；
- 使用真鼓、电子鼓、哑鼓垫练习；
- 记录个人目标与练习过程；
- 后续扩展 MIDI 准确度反馈。

尤其值得肯定的是，文档明确当前阶段以军鼓节奏训练为核心，但底层节奏型设计不绑定军鼓。这为后续扩展套鼓、Groove、MIDI 评分留下了空间。

---

### 3.2 非目标范围设置合理

PRD 中明确 MVP 暂不做以下内容：

- 手机原生 App；
- 云同步；
- 账号系统；
- 老师端；
- AI 鼓谱转写；
- 完整套鼓谱编辑；
- 麦克风识别；
- MIDI 准确度评分；
- 游戏化排行榜；
- 商业曲库。

这组非目标非常重要。

它能有效防止 MVP 在第一阶段被复杂功能拖垮。SnareLab 的第一阶段核心不是“什么都能做”，而是形成一个可以每天使用的练习闭环。

---

### 3.3 响应式策略正确

文档采用的是 **Responsive Web 设计**，而不是单独设计手机端。

这是正确判断。

SnareLab 的核心使用场景应该是：

| 设备 | 定位 |
|---|---|
| 桌面浏览器 | 编排、管理、记录 |
| 平板横屏 | 核心练习场景 |
| 平板竖屏 | 可用练习场景 |
| 手机 | 查看、轻量记录、临时编辑 |
| 微信小程序 | 后续可能形态，不进入 MVP |

产品视觉也保持了“深色练习驾驶舱”风格，没有因为响应式布局而切换到另一套视觉语言。这一点非常重要。

---

### 3.4 术语体系已经从“课堂”转向“个人练习”

文档已经明确避免使用：

- 课堂作业；
- 上课记录；
- 老师反馈；
- 教学后台。

并统一替换为：

- 个人目标；
- 练习笔记；
- 问题记录；
- 练习回顾。

这个调整让产品语义更加通用，也更贴近“个人练习系统”的长期方向。

---

### 3.5 技术栈合理

推荐技术栈为：

| 层级 | 技术 |
|---|---|
| 前端 | React + TypeScript + Vite |
| 状态管理 | Zustand |
| 本地数据库 | IndexedDB + Dexie |
| 音频 | Web Audio API |
| 播放调度 | Web Audio lookahead scheduler |
| 记谱 | VexFlow |
| MIDI | Web MIDI API，后续 |
| 桌面封装 | Tauri，后续 |

这套技术栈适合当前产品形态：

- 本地优先；
- 不依赖账号系统；
- 需要较强前端交互；
- 需要本地保存练习数据；
- 需要精确音频调度；
- 后续可扩展 MIDI 输入。

## 4. 主要问题与建议

## 4.1 MVP 范围存在轻微自相矛盾

### 问题

文档前半部分将“练习回顾”纳入 MVP 目标，但开发计划中又把完整练习回顾放在 Phase 5。

这会造成理解偏差：

- 如果 MVP 包含完整练习回顾，Phase 0 到 Phase 4 的范围就不够；
- 如果 MVP 只做到 Phase 0 到 Phase 4，那么练习回顾就不应被描述为 MVP 必须能力。

### 建议

将 MVP 中的“练习回顾”降级为“最小练习记录”。

建议改为：

```text
MVP 必须支持最小练习记录：
- 保存练习时间
- 保存练习 BPM
- 保存练习模式
- 保存使用的节奏型 / 队列
- 可选备注

完整练习回顾、主观评分、目标进展分析、本周趋势统计进入 Phase 5。
```

---

## 4.2 Phase 1 首页功能偏重

### 问题

Phase 1 中包含了：

- 本周进度；
- 当前 Focus；
- 下一步建议。

这些模块依赖后续练习数据。如果在数据尚未形成前过早实现，容易变成空洞仪表盘。

### 建议

Phase 1 保留：

| 功能 | 建议优先级 |
|---|---|
| 首页布局 | P0 |
| 快速入口 | P0 |
| 继续上次练习 | P0 |
| 当前个人目标 | P0 |
| 最近练习 | P0 |
| 新建练习笔记 | P0 |

Phase 5 再实现：

| 功能 | 建议阶段 |
|---|---|
| 本周进度 | Phase 5 |
| 当前 Focus | Phase 5 |
| 下一步建议 | Phase 5 或 Phase 6 |

---

## 4.3 RhythmPattern 数据模型需要增强

### 问题

当前 `RhythmPattern` 数据模型适合列表展示，但不足以支撑以下能力：

- 播放调度；
- VexFlow 渲染；
- 三连音 tuplet；
- 圆点辅助显示；
- 后续 MIDI 评分；
- 套鼓映射。

当前模型中缺少：

- `code`；
- `slotLabels`；
- `hitSlots`；
- `restSlots`；
- `ui`；
- `render`；
- VexFlow atoms 描述。

### 建议

将 `RhythmPattern` 扩展为以下结构：

```ts
type RhythmPattern = {
  id: string
  code: string
  name: string
  displayNameZh: string
  subdivision: 'quarter' | 'eighth' | 'sixteenth' | 'triplet'
  slotLabels: string[]
  steps: boolean[]
  hitSlots: number[]
  restSlots: number[]
  hitCount: string
  difficulty: 1 | 2 | 3 | 4 | 5
  tags: string[]

  ui: {
    cardFallbackGlyph: string
    dotStates: ('hit' | 'rest')[]
  }

  render: {
    engine: 'vexflow'
    staff: 'percussion-one-line'
    atoms: Array<{
      slotIndex: number
      kind: 'note' | 'rest'
      duration: string
    }>
    beamGroup?: boolean
    tuplet?: 'triplet' | null
  }

  createdAt?: string
  updatedAt?: string
}
```

核心原则：

```text
视觉渲染、播放逻辑、练习生成、后续 MIDI 评分，必须读取同一份 RhythmPattern 数据。
```

---

## 4.4 需要明确节奏型数据主源

### 问题

文档目前写了要建立 RhythmPattern 数据结构，并内置十六分和三连音节奏型，但没有明确：

- TypeScript Catalog 是主数据源；
- JSON Catalog 是主数据源；
- 两者如何同步。

如果不提前规定，后续很容易出现 TS 与 JSON 双账本问题。

### 建议

在技术方案中新增一节：

```text
节奏型数据主源策略：

开发期以 `frontend/src/catalog/rhythm-pattern-catalog.v1.ts` 作为唯一主数据源。
`frontend/src/catalog/rhythm-pattern-catalog.v1.json` 由脚本自动生成，不手工维护。
播放逻辑、VexFlow 渲染、练习生成、后续 MIDI 评分均读取统一 RhythmPattern schema。
```

并增加脚本约定：

```bash
pnpm generate:rhythm-json
```

## 4.5 缺少 PracticeExercise 实体

### 问题

文档中已有：

- `PracticeQueue`：练习队列；
- `PracticeSession`：一次练习记录；
- `PersonalGoal`：个人目标；
- `PracticeNote`：练习笔记。

但练习库 Library 中定义了多个练习来源：

- preset；
- generated；
- manual；
- goal；
- favorite。

这些“可复用练习”目前缺少一个稳定承载对象。

`PracticeQueue` 更像临时队列，`PracticeSession` 是练习记录，两者之间应该增加一个 `PracticeExercise`。

### 建议

新增数据模型：

```ts
type PracticeExercise = {
  id: string
  title: string
  source: 'preset' | 'generated' | 'manual' | 'goal' | 'favorite'
  queue: PracticeQueue
  linkedGoalIds: string[]
  tags: string[]
  createdAt: string
  updatedAt: string
}
```

这样 Library 才有明确的数据实体。

## 4.6 跟练页 Phase 4 可以拆分

### 问题

Phase 4 当前包含：

- 谱面渲染；
- 播放控制；
- Count-in；
- 拍位高亮；
- 小节高亮；
- 红色播放线；
- 节拍器声音；
- 鼓点示范；
- 声音层开关；
- BPM 显示；
- BPM 调节；
- BPM 拨盘；
- 自动保存练习信息。

这些都是合理能力，但一次性实现略重。

### 建议

拆成 Phase 4A 与 Phase 4B。

#### Phase 4A：最小可练版本

```text
- 渲染当前练习谱面
- 播放 / 暂停 / 重置
- Count-in
- 节拍器
- BPM 显示
- BPM +1 / -1
- 当前拍位高亮
- 保存练习记录
```

#### Phase 4B：体验增强版本

```text
- 红色播放线
- 当前小节高亮
- 鼓点示范声音
- 声音层开关
- BPM 拨盘
- 自动保存
```

---

## 4.7 长按编辑需要明确不属于 MVP P0

### 问题

PRD 中写了跟练页支持长按音符进行局部编辑，包括：

- 改为休止；
- 添加 / 删除重音；
- 设置 R / L；
- ghost note；
- 删除音符；
- 拆分节奏。

这些功能非常有价值，但不应该让开发者误解为 MVP 必须完成。

### 建议

在对应章节增加说明：

```text
长按编辑属于 Phase 6 自定义增强能力，不属于 MVP P0。
当前章节中的 P1/P2 表示 Phase 6 内部优先级，而非全局 MVP 优先级。
```

---

## 4.8 缺少验收标准

### 问题

当前文档已经有功能列表和开发计划，但缺少可执行的验收标准。

这会导致开发时出现：

- 功能是否完成难以判断；
- Phase 是否可以关闭难以判断；
- MVP 是否可用难以判断。

### 建议

为每个核心模块增加 Acceptance Criteria。

#### Rhythm 页面验收标准

```text
1. 用户可以看到十六分与三连音节奏型。
2. 用户可以多选节奏型。
3. 用户可以切换随机 / 顺序模式。
4. 用户可以生成练习队列。
5. 用户点击开始后进入 Trainer 页面。
```

#### Trainer 页面验收标准

```text
1. 用户可以看到当前练习谱面。
2. 用户可以开始、暂停、重置播放。
3. 用户可以调整 BPM。
4. 节拍器声音与视觉高亮基本同步。
5. 用户结束后可以保存练习记录。
```

#### Log 页面验收标准

```text
1. 用户可以看到最近练习记录。
2. 用户可以新建练习笔记。
3. 用户可以查看练习详情。
4. 用户可以按时间倒序查看历史。
```

## 6. 推荐的 MVP 表述

建议将 V1.0 MVP 目标统一改为：

```text
V1.0 MVP 目标：

完成一个可每天使用的本地优先军鼓节奏练习闭环。

用户可以：
1. 打开首页查看当前目标和最近练习。
2. 选择十六分 / 三连音基础节奏型。
3. 生成随机或顺序练习队列。
4. 进入跟练页看谱练习。
5. 调整 BPM。
6. 听节拍器和鼓点示范。
7. 保存最小练习记录。
8. 新建个人目标和练习笔记。

暂不实现：
- MIDI 准确度评分
- 完整练习回顾系统
- 套鼓谱编辑
- 云同步
- 手机原生 App
- 微信小程序
- AI 转写
```

## 7. 推荐的 MVP Definition of Done

建议增加以下 MVP 完成定义：

```text
当以下条件全部满足时，SnareLab V1.0 MVP 可视为完成：

1. 用户可以从首页进入节奏型选择页。
2. 用户可以选择至少 24 个基础节奏型中的任意组合。
3. 用户可以生成随机或顺序练习队列。
4. 用户可以进入跟练页看到谱面。
5. 用户可以启动、暂停、重置练习播放。
6. 用户可以调整 BPM。
7. 用户可以听到节拍器和鼓点示范。
8. 用户可以保存一条最小练习记录。
9. 用户可以在日志页查看最近练习记录。
10. 用户可以新建个人目标和练习笔记。
11. 所有数据保存在 IndexedDB 中，刷新页面后不丢失。
12. 桌面和平板浏览器布局可用。
```

## 10. 一句话总结

SnareLab 当前 PRD 的方向是正确的：它已经从“想做一个练鼓软件”进化成了“本地优先、深色驾驶舱风格、以军鼓节奏训练为起点的个人架子鼓练习系统”。

下一步不是继续加功能，而是把 MVP 边界压紧，把数据模型打牢，把验收标准补齐。这样第一版才会像一只调好张力的小军鼓：清楚、利落、每天都愿意敲。
